# Product Brief: AI 上下文工程可视化项目

## Executive Summary

**AI 上下文工程可视化项目**是一个 **AI 有机体构建平台**——不是编排 AI 的"动作"，而是让用户设计 AI 的"存在"。

**核心洞察：** AI 本质无状态。是精密的上下文工程设计让 AI 有了状态——让 AI Agent 真正"活"了起来。

**我们构建的是：** 一个通用的 AI 上下文工程平台（高度模块化底层）+ 一套小说创作官方模板（验证场景），遵循"一花一世界"的设计哲学——以具体场景为锚，以通用机制为本。

**核心信念：** 高度模块化底层 + 高度用户自定义 = 立于不败之地。

---

## Core Vision

### Problem Statement

**AI 本质无状态，而现有工具只能编排它的"动作"——市场缺少一个能让用户设计 AI 的"存在"的平台：通过上下文工程赋予 AI 状态，构建一个有器官、有神经系统、能持续运转的 AI 有机体。**

### Problem Impact

这个问题在实际场景中的具体表现：

| 场景 | 痛点 |
|------|------|
| **长篇小说创作** | 角色性格前后矛盾、设定遗忘、时间线混乱——AI 无法"记住"已建立的世界 |
| **游戏世界构建** | NPC 行为不一致、任务逻辑断裂——AI 无法感知整体状态 |
| **企业知识管理** | 信息更新后关联内容不同步——缺乏自动化的变更传播 |
| **复杂项目协作** | 人机协同停留在"人发指令、AI 执行"——AI 无法作为持续参与的成员 |

**根本原因：** 现有工具把 AI 当作"执行节点"，而非"活着的存在"。

### Why Existing Solutions Fall Short

| 现有方案 | 它们做了什么 | 它们缺少什么 |
|---------|-------------|-------------|
| **Dify / LangChain** | 节点连线、工作流编排 | 无状态感知、无变更传播、AI 是被动节点 |
| **RAG 知识库** | 静态检索、向量匹配 | 无动态状态管理、无上下文生命周期 |
| **自定义 Agent 框架** | 代码级灵活性 | 开发门槛高、缺乏可视化、难以复用 |
| **专用领域工具** | 特定场景功能 | 功能固化、无法迁移到其他场景 |

**核心差距：** 没有平台能让用户系统化地设计、管理和可视化 AI 的上下文生命系统。

### Proposed Solution

构建一个 **AI 有机体构建平台**，核心特征：

**1. 上下文工程作为核心**
- 统一抽象：Context Unit（上下文单元）= 变量 + 卡片
- 四类边关系：Reference / Activation / Containment / Derivation
- 变更传播：神经系统式的联动响应

**2. 有机体范式（非工作流范式）**
- 用户不是"编排流程"，而是"设计有机体"
- 可视化蓝图：卡片=器官，边=神经/血管，传播=神经信号

**3. AI Agent 作为"系统员工"**
- 深度嵌入系统，持续感知、响应、协同
- 按 Context Unit 责任划分，而非按任务分类

**4. 人机协同围绕状态**
- 人和 AI 共同管理项目中的状态（变量）
- 四种确认模式：严格 / 宽松 / 批量接受 / YOLO

**5. 生态兼容与深度融合**
- **全面兼容** MCP 工具和 Skills 设计范式
- **深度融入** 这些生态理念到系统内 AI Agent 设计
- **策略意图**：利用大模型已有的工具调用训练数据，让 Agent 在系统中"如鱼得水"

### Success Vision 成功画面

当用户使用这个平台时，他们能感知到：

> **一个"活着的系统"——每个变量（器官）之间的关联清晰可见，变更沿着"神经系统"自动传播，AI Agent 像系统内部的员工一样持续感知和响应。整个有机体在呼吸、在运转。**

### Key Differentiators

| 差异化维度 | 本项目 | 现有方案 |
|-----------|--------|---------|
| **核心范式** | 设计 AI 的"存在"（有机体） | 编排 AI 的"动作"（工作流） |
| **状态管理** | 系统化上下文工程，赋予 AI 真正的状态 | prompt + workflow，本质无状态 |
| **人机协同** | 围绕项目状态（变量）共同管理 | 人发指令，AI 执行 |
| **AI Agent 角色** | 系统内部的"员工"，深度协同 | 可调用的工具/节点 |
| **变更传播** | 有机联动，神经系统式响应 | 无，或需手动触发 |
| **可视化** | 有机系统蓝图，可感知"活着的系统" | 工作流图/节点图 |
| **生态策略** | 兼容 MCP/Skills + 深度融入系统理念 | 各自独立或简单集成 |

**方法论优势——"一花一世界"：**
- 以中文长篇网络小说创作为锚点场景（花）
- 每个解决方案都设计为通用机制（世界）
- 既避免过度设计，又保证平台普适性
